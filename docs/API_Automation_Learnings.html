1 · Project overview & architecture

A pytest-based REST API automation framework in Python using the requests library. It tests two live APIs:

  • JSONPlaceholder (https://jsonplaceholder.typicode.com) — a public fake REST API, no auth. Used for CRUD + schema + data-driven tests.
  • Restful-Booker (https://restful-booker.herokuapp.com) — a token-authenticated booking API. Used for the auth tests, because it actually enforces the token (rejects unauthenticated mutations).

Request flow

Test  →  fixture (APIClient)  →  requests.Session  →  external API  →  assertions
                                                                       (status + JSON body + schema)

2 · Guiding principle — tests must fail loudly

The single rule running through every decision below: a test should fail loudly when something is wrong, never pass quietly. The worst outcome in automation is a green test that verified nothing — it gives false confidence and hides real regressions.

Every design choice in this project serves that principle: raising on a missing schema, --strict-markers, explicit encoding, per-case test IDs, the negative auth test. Each one favours an obvious, immediate failure over a silent, misleading pass.

Interview"What's the worst kind of test?" → A test that passes without actually checking anything (no assertion, an empty schema that matches everything, a mistyped marker that's silently skipped). Always design so mistakes surface as failures, not as false greens.

3 · The APIClient wrapper

Rather than scatter raw requests.get(...) calls across tests, a thin wrapper centralises the base URL, a shared session, logging, and headers.

class APIClient:
    def __init__(self, base_url: str, headers: dict = None):
        if headers is None:                       # NOT headers={} — see footgun below
            headers = {"Content-Type": "application/json; charset=UTF-8"}
        self.base_url = base_url
        self.session = requests.Session()
        self.session.headers.update(headers)

    def get(self, endpoint, headers=None):
        url = f"{self.base_url}{endpoint}"
        logger.info(f"GET {url}")
        response = self.session.get(url, headers=headers)
        logger.info(f"Response {response.status_code}: {response.text}")
        return response
    # post / put / patch / delete follow the same shape

Why a Session (not bare requests.get)?

  • Connection reuse — keeps a TCP connection alive across calls (faster).
  • Header persistence — set auth/content-type once, applied to every request.
  • One place to log — every request/response is logged uniformly.
FootgunMutable default argument. Writing def __init__(self, headers={...}) is a classic Python bug: the default dict is created once and shared across all calls — mutating it leaks state between instances. Fix: default to None and assign a fresh dict inside.

4 · Fixtures & conftest.py

A fixture is reusable setup/teardown that pytest injects into a test by matching the parameter name. conftest.py holds fixtures shared across all test files (no import needed — pytest auto-discovers it).

@pytest.fixture
def api_client_jsonplaceholder():
    api_client = APIClient(base_url=API_ENDPOINT_JSONPLACEHOLDER)
    yield api_client                 # <-- test runs here
    api_client.session.close()       # <-- teardown after the test

The yield splits setup (before) from teardown (after) — the session is always closed, even if the test fails.

Fixture scope — the key decision

ScopeCreated…Use when
function (default)once per teststate must be fresh/isolated per test (e.g. a booking to delete)
sessiononce per whole runexpensive + safe to share read-only (e.g. an auth token)
Fresh instance per test method. pytest creates a new instance of the test class for every test method. So self.x = ... set in one test is gone in the next — sharing state via class attributes silently fails. The correct way to pass data between tests is a fixture (e.g. created_booking), which also removes hidden ordering dependencies.

5 · JSON Schema validation

The problem it solves

Asserting a single value (response.json()["id"] == 1) only confirms this one record looks right. It misses a whole class of contract regressions: a field renamed (userId → user_id), a type changing (id becoming the string "1"), a field disappearing, or an unexpected null. Checking every field's type by hand on every test is verbose and brittle.

The schema (the contract, written as JSON)

{
  "type": "object",
  "properties": {
    "userId": { "type": "integer" },
    "id":     { "type": "integer" },
    "title":  { "type": "string" },
    "body":   { "type": "string" }
  },
  "required": ["userId", "id", "title", "body"]
}

How it's used

from jsonschema import validate
from utils.data_loader import load_schema

response = api_client_jsonplaceholder.get("/posts/1")
assert response.status_code == 200                    # status FIRST
validate(response.json(), load_schema("post_jsonplaceholder"))   # then structure
assert response.json()["userId"] == 1                 # then business value

Example response being validated

GET https://jsonplaceholder.typicode.com/posts/1
Response 200:
{
  "userId": 1,
  "id": 1,
  "title": "sunt aut facere repellat provident occaecati...",
  "body":  "quia et suscipit\nsuscipit recusandae..."
}

validate() returns silently on a match and raises ValidationError (failing the test, e.g. "'userId' is a required property") on a mismatch.

Order matters. Validate after the status assertion — so a 404 fails clearly as "wrong status", not as a confusing schema error about the error body.

Key keywords

KeywordMeaning
typestring/integer/number/boolean/object/array/null
propertiesdeclares each field's sub-schema
requiredlist of fields that must be present
itemsschema each element of an array must satisfy (for list endpoints like GET /posts)
additionalProperties:falsestrict — reject any undeclared field (more rigorous, more brittle)

Object vs. array: /posts/1 returns an object → validate against an object schema. /posts returns a list → use {"type":"array","items":{ <object schema> }} so every element is checked.

Interview"Value assertions vs schema validation?" → Use both. Schema guards the contract/structure (types, required fields); value assertions confirm business logic (the right data). Together this is contract testing.

6 · File encoding — why encoding="utf-8" is explicit

A file on disk is just bytes; an encoding is the rulebook mapping characters ↔ bytes. The same character maps to different bytes under different encodings (é is two bytes in UTF-8, one in Windows-1252).

open() without encoding= uses a platform-dependent default: Windows often defaults to cp1252, while Linux and CI default to UTF-8. That asymmetry is the bug — code behaves differently across environments, producing the classic "works on my machine / passes locally, fails in CI" trap, or UnicodeDecodeError the moment a smart-quote, em-dash, accented name, or emoji appears in a data file.

with open(path, "r", encoding="utf-8") as f:   # always explicit
    return json.load(f)
Rule: always pass encoding="utf-8" when opening text files. UTF-8 represents all Unicode, is ASCII-backward-compatible, and is the JSON spec default. (Related but separate: ensure_ascii=False when writing JSON controls whether non-ASCII is escaped to \uXXXX — not needed for reading.)

7 · Robust path handling (data_loader.py)

Loading by relative path (open("data/schemas/x.json")) resolves against the current working directory — wherever pytest happened to be launched — so it breaks when run from a subfolder or in CI. The fix is to anchor to the file's own location:

from pathlib import Path

PROJECT_ROOT = Path(__file__).resolve().parent.parent   # walk up from utils/ to root
DATA_DIR     = PROJECT_ROOT / "data"
SCHEMA_DIR   = DATA_DIR / "schemas"

def load_schema(name: str) -> dict:
    with open(SCHEMA_DIR / f"{name}.json", "r", encoding="utf-8") as f:
        return json.load(f)

def load_data(name: str):                                # used by parametrize
    with open(DATA_DIR / f"{name}.json", "r", encoding="utf-8") as f:
        return json.load(f)
  • Path(__file__).resolve() → absolute path to the module itself
  • .parent.parent → walk up from utils/ to the project root
Fail-loud detail. An earlier version returned an empty {} schema for unknown names. An empty schema validates anything, so a typo'd schema name would silently pass. Building the path from the name instead means a missing file raises FileNotFoundError — a loud, correct failure.

8 · Pytest markers (smoke, regression, auth)

A marker is a label on a test (@pytest.mark.smoke). It carries no behaviour by itself — it just tags the test so groups can be selected or skipped.

@pytest.mark.smoke
@pytest.mark.regression
def test_get_post(self, api_client_jsonplaceholder):
    ...

Why — run subsets at different cadences

  • smoke = tiny, fast critical-path subset → run on every commit/PR for quick feedback.
  • regression = the full, slower suite → run nightly / pre-release.

A test can carry multiple markers; smoke tests are usually also regression (smoke is a subset).

Selecting

pytest -m smoke
pytest -m "smoke or regression"
pytest -m "regression and not smoke"

The division of labour

PieceJob
@pytest.mark.smoke (decorator)Tags the test
pytest -m smoke (CLI flag)Selects tagged tests
pytest.ini markers =Declares the name is legit + documents it
--strict-markers (set in pytest.ini) upgrades an unregistered/mistyped marker from a silent warning to a hard error at collection time. Without it, a typo like @pytest.mark.smoek is silently accepted as a brand-new marker, so pytest -m smoke quietly skips that test — a hidden coverage hole. Fail-loud, applied to markers.

9 · Data-driven tests (@pytest.mark.parametrize)

Runs the same test body with different inputs, and crucially each input becomes a separate test — its own pass/fail line.

@pytest.mark.parametrize(
    "case",                                    # 1. param name injected into the test
    load_data("post_cases"),                   # 2. list of cases (resolved at collection time)
    ids=lambda c: f"post{c['post_id']}-status{c['expected_status']}",  # 3. readable labels
)
def test_get_post_cases(self, api_client_jsonplaceholder, case):
    response = api_client_jsonplaceholder.get(f"/posts/{case['post_id']}")
    assert response.status_code == case["expected_status"]
    if case["expected_status"] == 200:
        validate(response.json(), load_schema("post_jsonplaceholder"))
        assert response.json()["userId"] == case["expected_user"]

The data file (data/post_cases.json)

[
  { "post_id": 1,    "expected_status": 200, "expected_user": 1 },
  { "post_id": 2,    "expected_status": 200, "expected_user": 1 },
  { "post_id": 11,   "expected_status": 200, "expected_user": 2 },
  { "post_id": 9999, "expected_status": 404, "expected_user": null }
]

Resulting test run — 4 independent tests

test_get_post_cases[post1-status200]      PASSED
test_get_post_cases[post2-status200]      PASSED
test_get_post_cases[post11-status200]     PASSED
test_get_post_cases[post9999-status404]   PASSED
The collection-time gotcha. parametrize is evaluated before any fixture runs — so the data source cannot be a fixture; it must be a plain function call (load_data(...)). The decorator needs a concrete list right then.
Why not a for loop inside one test? A loop stops at the first failure — you never learn whether later cases also broke, and the report shows one failed test instead of "1 of 4 failed." parametrize gives independent, individually-named tests. The ids= lambda makes a failure name the broken case (post9999-status404) instead of an opaque [case3].

10 · Token authentication & the auth fixture chain

Token auth is a two-step dance: (1) POST /auth with credentials → server returns a token; (2) send that token on every protected request. Restful-Booker expects it as a Cookie: token=<token> header (other APIs use Authorization: Bearer <token> — same idea, different header).

Step 1 — log in once (session-scoped)

@pytest.fixture(scope="session")
def restful_booker_token():
    client = APIClient(base_url=API_ENDPOINT_RESTFULBOOKER)
    response = client.post("/auth", {"username": "admin", "password": "password123"})
    return response.json()["token"]
POST https://restful-booker.herokuapp.com/auth | body: {'username':'admin','password':'password123'}
Response 200: {"token":"6cf5ed36614a18c"}

Why session scope? The token stays valid for the whole run — logging in once is faster, and it's read-only shared state (safe to share). Function scope would re-login per test for no benefit.

Step 2 — build an authenticated client (two-layer design)

@pytest.fixture(scope="session")
def api_session_restfulbooker(restful_booker_token):
    headers = {
        "Content-Type": "application/json",
        "Accept": "application/json",
        "Cookie": f"token={restful_booker_token}",   # token baked into every request
    }
    client = APIClient(base_url=API_ENDPOINT_RESTFULBOOKER, headers=headers)
    yield client
    client.session.close()
Separation of concerns. restful_booker_token gets credentials (returns the raw token); api_session_restfulbooker builds an authed client (depends on the token). Tests just call methods — the token attaches itself. This mirrors real frameworks and keeps the raw token reusable for negative tests.

The negative test — the most valuable auth assertion

@pytest.mark.auth
def test_delete_booking_withouttoken(self, created_booking):
    client = APIClient(base_url=API_ENDPOINT_RESTFULBOOKER)   # NO token
    response = client.delete(f"/booking/{created_booking}")
    assert response.status_code == 403
DELETE https://restful-booker.herokuapp.com/booking/2962
Response 403: Forbidden
This is the point. A DELETE with no token → 403 proves auth is actually enforced, not just that login returns a string. (ReqRes was rejected as a target precisely because its token isn't enforced on anything — you could never write this test.) Pair it with the positive twin (test_delete_booking, same call with token → success) for the with/without contrast.

11 · Response-shape gotcha — create vs. update nest differently

The same API can wrap responses differently per endpoint. Restful-Booker is a perfect example:

Create — wrapped under "booking"

POST /booking
Response 200:
{
  "bookingid": 2952,
  "booking": {
    "firstname": "Suman",
    "lastname":  "Mandal",
    "totalprice": 111,
    "depositpaid": true,
    "bookingdates": { "checkin": "2018-01-01", "checkout": "2019-01-01" }
  }
}
assert response.json()["booking"]["firstname"] == "Suman"   # nested

Update (PATCH) — returned at the top level

PATCH /booking/2956 | body: {'firstname':'YoYo','lastname':'Mandal'}
Response 200:
{
  "firstname": "YoYo",
  "lastname":  "Mandal",
  "totalprice": 111,
  "depositpaid": true,
  "bookingdates": { "checkin": "2018-01-01", "checkout": "2019-01-01" }
}
assert response.json()["firstname"] == "YoYo"               # NOT nested
Lesson. Never assume one response shape across endpoints. Always read the actual body — the create result nests under ["booking"], but PATCH returns the booking object directly. A wrong path here is a KeyError at runtime.

12 · The full CRUD chain & "trust but verify"

create → update → delete, each test self-contained via the created_booking fixture (no cross-test ordering).

OperationEndpointStatusNotes
CreatePOST /booking200no token needed; result nested under booking
UpdatePATCH /booking/{id}200token required; result at top level
Delete (with token)DELETE /booking/{id}201token required; returns 201 Created (quirk!)
Delete (no token)DELETE /booking/{id}403negative test — proves enforcement

"Trust but verify" — confirm the side effect with an independent read

A mutation test is strongest when a separate read confirms the change actually persisted — don't trust the mutation endpoint's own echo.

# after PATCH → re-read and assert the new value stuck
PATCH /booking/1668   Response 200: {"firstname":"YoYo",...}
GET   /booking/1668   Response 200: {"firstname":"YoYo",...}   ✓ persisted

# after DELETE → re-read and assert it's gone
DELETE /booking/1687  Response 201: Created
GET    /booking/1687  Response 404: Not Found              ✓ deleted
Setup fixture vs. test under test. created_booking and the create test both call POST /booking — but that's not duplication to remove. The test's job is to verify creation (it asserts); the fixture's job is to arrange a precondition for delete/update (it doesn't assert). What is worth de-duplicating is the request data — hence the shared data/booking_payload.json. General rule: test code tolerates more duplication than production code, but centralise shared data.

14 · Rapid-fire interview Q&A

QWhy wrap requests in an APIClient class instead of calling it directly?
Centralises base URL, shared session (connection reuse + persistent headers), and logging. One place to change, consistent behaviour, DRY tests.

Qfunction vs session fixture scope?
function = fresh per test (isolation — e.g. a booking to delete). session = once per run (expensive + safe to share read-only — e.g. an auth token).

QHow do you share data between tests?
A fixture (e.g. created_booking), not class attributes — pytest makes a fresh class instance per test, so self.x doesn't carry over.

QValue assertion vs schema validation?
Both. Schema checks structure/types/required fields (contract); value assertions check business logic (right data). Together = contract testing.

QWhy encoding="utf-8" everywhere?
open()'s default is platform-dependent (Windows cp1252 vs Linux UTF-8). Being explicit avoids "passes locally, fails in CI" and Unicode errors on special characters.

QWhat does --strict-markers do and why?
Turns an unregistered/mistyped marker into a collection error instead of a silent warning — so a typo can't quietly skip a test.

QWhy can't parametrize data come from a fixture?
parametrize is resolved at collection time, before fixtures run — so it needs a plain function call returning a concrete list.

QMost valuable auth test?
The negative one: a protected call without a token must return 403 — it proves auth is enforced. Pair with a positive (with-token) test for contrast.

QHow do you handle token auth across many tests?
Two-layer fixtures: one logs in once (session-scoped) and returns the token; another builds an authenticated client with the token in its headers. Tests just use the client.

QWhat's "trust but verify" in API testing?
After a mutation, do an independent read to confirm the side effect actually persisted (PATCH → GET shows new value; DELETE → GET returns 404) — don't trust the mutation's own echoed response.

QTests are flaky against a live third-party API — what do you do?
Distinguish a real failure from environment flakiness (re-run; here Restful-Booker's free Heroku instance intermittently 5xx'd). Long-term: retry on transient network errors, or mock/stub the external service for deterministic CI.
